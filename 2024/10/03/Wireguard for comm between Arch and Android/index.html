Wireguard for comm between Arch and Android

2024-10-03 Arch Linux android, arch, chatgpt history, wireguard 0 Comments Word Count: 765(words) Read Count: 4(minutes)

Build a wiregard communication between the Arch laptop and my Android phone. Following the instructions from ChatGPT, which I proved to work.

https://wiki.archlinux.org/title/WireGuard#

From GPT:

To set up a private network between your Arch Linux laptop and your Android phone within a large LAN, you can create a secure tunnel or virtual private network (VPN) to ensure only these two devices communicate directly, safely bypassing other potentially malicious devices. Here’s a simple approach:

Option 1: WireGuard VPN

WireGuard is a fast and modern VPN protocol that’s well-suited for creating a private network between your devices.

Can also use other tools, as long as the Wireguard is supported. For example, the network manager from KDE does support Wireguard, so just configure it in the same way.

On Android (Phone):

  1. Install the WireGuard app from the Play Store.

  2. Generate keys on the phone using the WireGuard app.

  3. Configure WireGuard on the phone:

    • Add a new tunnel in the app and set the following details:
      • Interface: Use your phone’s private key and set 10.0.0.2/24 as the IP.
      • Peer: Set the laptop’s public key and Allowed IPs: 10.0.0.1/32.
      • Endpoint: Your laptop’s LAN IP address with port 51820.

  4. Activate the tunnel in the WireGuard app.

Now, the devices can communicate securely over this private network.

You’re correct! In WireGuard, peers don’t automatically discover each other on a network without at least one side specifying an endpoint. WireGuard is a stateless, point-to-point VPN, so it relies on manually configuring endpoints to establish a connection between devices.

Here’s how it works:

Endpoint Role:

The endpoint is essentially the “address” where one device (peer) will attempt to contact the other. Without an endpoint, the peers won’t know where to send the initial packets.

Why You Need an Endpoint on at Least One Side:

  1. WireGuard doesn’t broadcast or discover peers like other networking protocols (e.g., DHCP). Both devices remain silent until one side initiates communication by sending a packet to the other’s IP address and port.
  2. At least one side needs the endpoint of the other peer (IP address and port) to start the handshake.

So, in your case where the Android phone and the Arch Linux laptop are on the same LAN, here’s how you should approach it:

Solutions for Your Case:

  1. Configure Endpoint on One Device:

    • You can set up the endpoint on one side (either the laptop or the phone). If you configure the endpoint on the laptop, it will initiate the connection to the phone, and vice versa.
    • For instance, if your phone has a static IP on the LAN (e.g., 192.168.1.10), you can configure the endpoint on the laptop to point to the phone.

  2. Configure Endpoint on Both Devices:

    • To ensure both devices can initiate the connection (especially useful if either device is mobile and changes networks frequently), you can configure endpoints on both sides.
    • For example, on your laptop, set the phone’s LAN IP as the endpoint, and on the phone, set the laptop’s LAN IP as the endpoint.

    Example (Android-side configuration):

  3. Dynamic IPs (No Static Endpoint): If either the phone or the laptop has a changing IP (e.g., on different networks):

    • You can configure a dynamic DNS (DDNS) service to ensure the device’s endpoint is always updated, even if the IP changes. This is helpful for devices that are frequently on different networks.
    • Otherwise, you’d need to update the endpoint manually each time the IP changes.

Summary:

  • Endpoint is needed for initial communication: At least one device must know the other’s IP and port (via the endpoint) to initiate a connection.
  • Best Practice: Configure an endpoint on one side, and optionally on both sides for more flexibility.
